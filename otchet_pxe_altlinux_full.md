# Отчёт по выполненной работе

## Тема работы

Развёртывание сервера централизованной сетевой установки операционных систем ALT Linux с использованием технологий PXE, DHCP, TFTP, NFS и HTTP.

---

# Цель работы

Организовать централизованную автоматизированную установку нескольких операционных систем ALT Linux по сети без использования физических носителей информации.

---

# Задачи работы

В рамках выполнения практического задания требовалось:

1. Настроить сетевую загрузку по PXE.
2. Развернуть DHCP и TFTP сервисы.
3. Подготовить установочные образы ALT Linux.
4. Реализовать выбор нескольких операционных систем из меню PXE.
5. Настроить автоматическую установку через файлы ответов.
6. Организовать передачу второго этапа установки по NFS.
7. Исследовать механизм работы загрузчика и установщика ALT Linux.
8. Устранить ошибки, возникающие при сетевой установке.

---

# Используемые технологии

В ходе работы использовались следующие сервисы и технологии:

- PXE (Preboot Execution Environment);
- DHCP;
- TFTP;
- NFS;
- HTTP (nginx);
- PXELINUX;
- bootchain ALT Linux;
- autoinstall.scm;
- initrd / initramfs.

# Настройка DHCP и PXE

Для организации сетевой загрузки был настроен сервис dnsmasq.

Сервис обеспечивал:

- выдачу IP-адресов клиентам;
- передачу параметров PXE;
- запуск загрузчика pxelinux.0;
- работу встроенного TFTP-сервера.

Был подготовлен отдельный конфигурационный файл для работы PXE-инфраструктуры.

В процессе настройки были исправлены ошибки привязки интерфейса и приведена к рабочему состоянию конфигурация DHCP.

---

# Настройка TFTP-сервера

Для хранения файлов сетевой загрузки был создан отдельный каталог:

    /srv/pxe

Структура каталога:

    /srv/pxe
    ├── pxelinux.0
    ├── pxelinux.cfg
    │   └── default
    └── images
        ├── workstation
        │   ├── vmlinuz
        │   └── initrd.img
        ├── kworkstation
        │   ├── vmlinuz
        │   └── initrd.img
        └── server
            ├── vmlinuz
            └── initrd.img

Для каждого дистрибутива были подготовлены собственные версии ядра Linux и initrd.

Проверена корректная передача файлов клиентам через TFTP.

---

# Создание PXE-меню

Был создан собственный конфигурационный файл:

    pxelinux.cfg/default

В меню реализован выбор между несколькими дистрибутивами:

- ALT Workstation;
- ALT KWorkstation;
- ALT Server.

Каждый пункт меню использовал:

- собственное ядро;
- собственный initrd;
- собственный каталог metadata;
- собственный источник второго этапа установки.

В результате была реализована полноценная мультизагрузочная среда PXE.

# Настройка HTTP-сервера

Для публикации файлов автоматической установки был настроен веб-сервер nginx.

Каталог metadata был организован следующим образом:

    /var/www/html/metadata
    ├── workstation-p11
    ├── kworkstation-p11
    └── server-p11

В каждом каталоге размещались файлы:

- autoinstall.scm;
- vm-profile.scm;
- install-scripts.tar.

Проверена доступность файлов через HTTP как с сервера, так и из среды установщика.

---

# Настройка автоматической установки

Для автоматической установки использовались параметры загрузки:

    ai curl=http://<server>/metadata/<profile>/

В процессе тестирования было установлено, что установщик ALT Linux ожидает путь к каталогу metadata, а не прямую ссылку на файл autoinstall.scm.

Первоначально использовался путь:

    curl=http://server/metadata/profile/autoinstall.scm

что приводило к ошибке получения metadata.

После исправления на:

    curl=http://server/metadata/profile/

автоматическая установка начала работать корректно.

---

# Настройка NFS-сервера

Для передачи второго этапа установки был настроен NFS-сервер.

Первоначально использовалась стандартная схема ALT Linux:

    /srv/public/netinst/current

Однако данное решение не позволяло одновременно использовать несколько установочных образов.

Для решения проблемы были подготовлены отдельные каталоги:

    /srv/public/netinst/workstation
    /srv/public/netinst/kworkstation
    /srv/public/netinst/server

В каждый каталог было помещено распакованное содержимое соответствующего установочного образа.

# Исследование механизма загрузки ALT Linux

Для определения механизма выбора установочного источника был выполнен анализ содержимого initrd.

В ходе исследования:

- выполнена распаковка initramfs;
- изучены сценарии bootchain;
- исследован механизм подключения NFS;
- проанализирована обработка параметров загрузки.

Было установлено, что по умолчанию bootchain использует каталог:

    /srv/public/netinst/current

Для поиска источника установки используются параметры:

    server
    directory

При их отсутствии используются значения по умолчанию.

---

# Анализ исходного кода bootchain

В процессе исследования был обнаружен механизм определения каталога второго этапа установки.

Логика работы:

1. Получение параметров server и directory.
2. Получение параметров через DHCP.
3. Использование стандартного каталога:

       /srv/public/netinst/current

4. Поиск файла второго этапа установки:

       live

Именно этот механизм вызывал ошибку:

    Second stage file not found: 'live'

при неправильной организации структуры установочного источника.

---

# Реализация выбора установочного образа

После анализа bootchain был найден рабочий способ указания каталога установки напрямую через параметры загрузки:

    automatic=method:nfs,server:192.168.56.10,directory:/srv/public/netinst/kworkstation

Аналогичные параметры были подготовлены для всех дистрибутивов.

Это позволило:

- отказаться от использования каталога current;
- использовать несколько образов одновременно;
- выбирать необходимый дистрибутив непосредственно из PXE-меню.

# Возникшие проблемы

## Ошибка получения metadata

Симптом:

Автоматическая установка не запускалась.

Причина:

Использовался путь к конкретному файлу autoinstall.scm вместо каталога metadata.

Решение:

Изменён параметр curl на путь к каталогу metadata.

---

## Ошибка второго этапа установки

Симптом:

    Second stage file not found: 'live'

Причина:

В качестве источника установки использовался файл образа, а не распакованное содержимое дистрибутива.

Решение:

Созданы отдельные каталоги с распакованным содержимым установочных образов.

---

## Проблема выбора нескольких образов

Симптом:

Все установки использовали один и тот же источник.

Причина:

bootchain по умолчанию использовал каталог:

    /srv/public/netinst/current

Решение:

Найден и применён параметр:

    directory=

позволяющий явно задавать каталог установки.
